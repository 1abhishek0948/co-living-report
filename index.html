Executive Summary

The repository now contains a working monorepo scaffold for a co-living marketplace. It includes a Next.js frontend, a Fastify API, a shared TypeScript domain package, a Prisma schema for the intended production data model, Docker-based local infrastructure, starter Terraform resources, and a mock-backed auth layer for account creation and login. The current runtime favors demonstrable product flows over full persistence, which is the correct tradeoff for a first delivery slice.

2. Technical Architecture

Primary Runtime Components
  • Web: Next.js 15 with React 19
  • API: Fastify 5 running via `tsx` in dev and start mode
  • Shared: workspace package for domain types and matching logic
  • Schema: Prisma 6 with PostgreSQL as target datastore
  • Local infra: PostgreSQL, Redis, MinIO
  • Deploy prep: ECS cluster, ECR, logs, S3 bucket in Terraform starter
Current Architectural Position

The application is intentionally in a modular-monolith phase. Domain boundaries exist already, but runtime persistence is still mocked in memory. This gives fast product iteration while preserving a clean path to replace repositories with Prisma-backed services. 

Browser
  -> Next.js web app (pages, auth session UI, discovery views)
      -> Fastify API
          -> in-memory domain store today
          -> Prisma schema and seed for DB-backed phase
          -> provider abstractions for Persona / Onfido / Stripe
      -> local storage session persistence for mock auth UI

Future production path:
Next.js / API -> PostgreSQL + Redis + object storage -> queues / workers -> managed cloud services

5. Matching Engine Detail

The matching engine lives in the shared package and is one of the strongest concrete pieces of business logic already implemented. It is deterministic, explainable, and returns both a score and a confidence value.

Dimension Weight Implementation Notes
Budget 25% Scores within-budget fully and decays when rent is outside target range
Timing 20% Matches listing availability to move-in window with 45-day decay outside range
Location 10% Compares listing area to preferred areas
Lifestyle 35% Cleanliness, noise, guests, sociability, sleep, WFH, smoking, pets
Household 10% Mean roommate fit with variance penalty to punish one severe mismatch

Hard blockers currently include:

  • Rent exceeding declared budget ceiling by more than 15%
  • No vacant slots
  • Required verification level not met
  • Listing rules conflict with smoking or pet status
  • Stay length outside listing min/max constraints

6. Data Model and Persistence Readiness

The runtime data store is still mocked for speed, but the Prisma schema is already modeling the intended production structure. That is important because it prevents a future rewrite of the domain shape.

Core relational entities
  • User and UserProfile
  • RoommatePreference and MoveInIntent
  • VerificationState and VerificationCase
  • Property, Unit, Listing, ListingOccupant
  • Application
  • Conversation, ConversationMember, Message
  • PaymentAccount and PaymentLedgerEntry
  • AuditLog
What this means operationally

The schema already supports owner inventory, seeker intent, trust state, household matching, chats, and financial audit trails. The missing work is repository replacement and transaction orchestration, not schema invention.

7. Auth, Trust, and Payments

The project now includes a basic sign-up and login flow. It is intentionally mock-backed and not yet suitable for production. The auth layer exists to unblock UI flows and stakeholder review, not to represent final security design.

  • Login and signup are implemented in the web app and backed by API routes.
  • Sessions are stored client-side in local storage for the current demo implementation.
  • Seeded demo logins exist for testing the mock experience.
  • KYC integration boundaries exist through Persona and Onfido provider wrappers.
  • Payments integration boundaries exist through a Stripe Connect-style provider wrapper.
Important trust note

The current account flow is a delivery scaffold. It does not yet include password hashing, CSRF protection, JWT/session signing, email verification, rate limiting, or server-backed session storage. Those belong in the next implementation phase.

10. Known Gaps and Risks

  1. The API runtime still uses in-memory state for listings, users, applications, chats, and auth. A server restart resets that data.
  2. The login flow is a demo scaffold, not production security. Passwords are not yet hashed and sessions are not yet persisted or signed on the server.
  3. Verification and payment providers are placeholders rather than live SDK-backed integrations.
  4. Prisma schema readiness is ahead of runtime repository implementation, which is the main remaining architecture gap.
  5. Terraform currently covers only starter cloud resources. Networking, databases, load balancing, WAF, and CD workflows still need expansion.

11. Recommended Next Phase

  1. Replace mock repositories with Prisma-backed services and transactions.
  2. Introduce proper auth with hashing, protected sessions, and route authorization.
  3. Persist chats, applications, and verification events in PostgreSQL.
  4. Wire real Persona / Onfido and Stripe SDK flows with signed webhooks.
  5. Add Redis-backed queues for notifications, recomputation, and moderation workflows.
  6. Expand Terraform into a deployable AWS environment with ECS services and managed data stores.

12. Final Assessment

This repository is in a strong scaffold state. It already demonstrates the product direction, architectural boundaries, matching logic, route shape, and UX surface expected for a co-living marketplace. The codebase is not yet a production system, but it is no longer an idea or a pure design artifact either. It is now an executable foundation with a clear path to persistence, security hardening, and cloud deployment.
